# Issue 6 — Task Analysis: Screen/UI Iteration

## Related Modules

- **`src/render.js`** — Complete rewrite of all rendering functions (court, net, player, ball, hud, menu, game_over)
- **`src/camera.js`** — Remove 3D perspective projection; replace with 2D coordinate mapping
- **`src/constants.js`** — Update screen dimensions, remove DEPTH_CHARS/FOV; add court ASCII layout constants
- **`src/main.js`** — Update `draw_game()` for new render API; camera init removed
- **`tests/`** — New tests for rendering functions

## Impacts

- **Rendering overhaul**: 3D perspective → 2D top-down ASCII court representation
- **Camera system removed**: No more `project()`, `project_char()`, `draw_line()`, `draw_rect()`, depth shading
- **Screen layout redesign**: New coordinate mapping from world coords to screen grid
- **Visual feedback added**: Shot indicators, ball trail/trajectory hints, serve direction indicator
- **Menu/game-over screens**: Keep but position within new layout

## Summary

Replace the unreadable first-person 3D perspective rendering with a clear 2D top-down ASCII court view inspired by Mario Tennis GBC. The court is drawn as ASCII art on a character grid. Players, ball, and HUD overlay on top. All game logic remains untouched.

### Phase 1: Tests
Write tests for the new rendering system:
- Court ASCII layout renders correct characters at expected positions
- HUD display renders correct score text
- Player character renders at mapped position
- Ball character renders at mapped position
- Menu screen renders
- Game over screen renders
- State overlays (serve prompt, point scored, shot type indicator)

### Phase 2: Constants & Layout
- Define new screen dimensions for 2D ASCII grid (e.g., 80×25 character cells)
- Define ASCII court template characters (court surface, lines, net)
- Define world-to-screen coordinate mapping formulas
- Remove FOV, DEPTH_CHARS, depth-related constants
- Remove camera.js `project()`, `draw_line()`, `draw_rect()` — no longer needed

### Phase 3: Core Render Logic
- Implement 2D coordinate mapping: world (x, z) → screen (col, row)
- Implement ASCII court drawing:
  - Court border/baselines/sidelines as `-`, `|`, `+` characters
  - Net as `=` or `#` across middle
  - Service boxes as inner lines
- Implement player rendering at mapped position with label char
- Implement ball rendering at mapped position with `o`
- Implement HUD overlay (score, games, sets)

### Phase 4: UI Polish & Integration
- Menu screen with difficulty selection
- Game over screen with winner + restart prompt
- Visual feedback: serve direction arrow, shot type notification, ball trail
- Update `main.js` `draw_game()` to use new render API
- Remove `camera.init()` from game loop
- Verify end-to-end gameplay: menu → serve → play → score → game over → restart
