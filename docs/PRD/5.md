# Issue 5 — Port ASCIITennis from TIC-80 to Web Platform

## Motivation

TIC-80 platform testing overhead is too high. Porting to web enables faster development, easier testing, and broader accessibility.

## Features

- Same ASCII tennis game running in a web browser
- WASD keyboard controls (W=up, A=left, S=down, D=right)
- Left mouse button for action (hit, serve, confirm)
- HTML5 Canvas-based rendering of ASCII graphics
- Game logic (physics, AI, scoring) preserved as-is

## Acceptance Criteria

- [ ] Game loads and runs in a modern web browser (Chrome, Firefox, Edge)
- [ ] Player moves with W/A/S/D keys (previously arrow keys)
- [ ] Player hits ball, serves, and confirms selections with left mouse click (previously B button)
- [ ] All 4 shot types work (flat, topspin, slice, lob) using mouse + movement keys
- [ ] Court, net, players, ball, HUD, and menus rendered as ASCII on canvas
- [ ] Game loop runs at 60fps via requestAnimationFrame
- [ ] AI opponent, scoring, state machine unchanged
- [ ] All TIC-80 APIs (TIC, btn, cls, print) eliminated

## Bug: CORS / Black Screen on file://

**Error:**
```
Access to script at 'file:///.../src/main.js' from origin 'null' has been blocked by CORS policy
'file:' URLs are treated as unique security origins
```

**Root Cause:**
`docs/DESIGN/5.md:113` specifies `<script type="module">` for ES6 modules with "no build step". However, browsers block ES module loads from `file://` protocol because `file:` URLs are treated as unique security origins. The module system requires an HTTP(S) server to set proper CORS headers.

**Impact:**
- Opening `index.html` directly via `file://` produces a black screen
- All game logic in ES modules fails to load
- Affects all browsers (Chrome, Firefox, Edge)

**Solution Options:**

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| A | Use local HTTP server + keep `<script type="module">` | Modern, no code change | Requires server command |
| B | Regular `<script>` tags (global scope) instead of modules | Works on file:// directly | Global namespace pollution |
| C | Inline all JS in index.html | Single file, works everywhere | Large file, hard to maintain |
| D | Bundle with Vite/Webpack | Dev server + build, modern | Adds build step |

**Recommended:** Keep ES modules for maintainability, but add a local dev server setup (`package.json` with `serve` or documented `python -m http.server` command). This preserves the module architecture while fixing the load issue.
