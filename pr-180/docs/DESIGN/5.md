# Issue 5 — Web Port: Architecture & Design

## Architecture

Monorepo HTML/JS web game. Game logic (court, camera, player, ball, ai, scoring, constants) translated 1:1 from Lua to plain JS modules. Platform layer (input, render, main) replaced with browser APIs.

```
index.html          → Entry point, canvas, script loading
src/
  constants.js      → Ported from constants.lua
  court.js          → Ported from court.lua
  camera.js         → Ported from camera.lua (project, draw_line → Canvas)
  player.js         → Ported from player.lua
  ball.js           → Ported from ball.lua
  ai.js             → Ported from ai.lua
  scoring.js        → Ported from scoring.lua
  input.js          → NEW: KeyboardEvent + MouseEvent → virtual btn states
  render.js         → NEW: Canvas fillText() replacing cls()/print()
  main.js           → NEW: requestAnimationFrame loop, state machine
```

## Data Flow

```
requestAnimationFrame
  → input.update()          (poll keyboard/mouse state → virtual buttons)
  → update_*()              (state machine: menu/serving/playing/point_scored/game_over)
  → draw_game()             (clear canvas, render court/net/players/ball/HUD)
```

## Key Design Decisions

### Canvas Rendering Strategy

- Replace `cls()` with `ctx.clearRect(0, 0, canvas.width, canvas.height)`
- Replace `print(str, x, y)` with `ctx.fillText(str, x * CHAR_W, y * CHAR_H)`
- Use monospace font (e.g., `"Courier New", monospace`) at a fixed size
- Character cell: `CHAR_W = 6`, `CHAR_H = 12` (or proportional to canvas size)
- All TIC-80 pixel coords (240×136) map to char-cell coords on canvas
- Screen logical size: 240×136 characters (TIC-80 resolution) mapped to pixels via scale factor 4 → 960×544 canvas

### Input Mapping

| TIC-80 Button | Web Equivalent |
|---|---|
| BTN_UP (idx 0) | Key W |
| BTN_DOWN (idx 1) | Key S |
| BTN_LEFT (idx 2) | Key A |
| BTN_RIGHT (idx 3) | Key D |
| BTN_A (idx 4) | Left mouse click |
| BTN_B (idx 5) | Left mouse click (same) |

- `keydown` → set virtual btn state to true
- `keyup` → set virtual btn state to false
- `mousedown` → set BTN_B (and BTN_A) to true
- `mouseup` → set BTN_B (and BTN_A) to false
- `input.update()` copies curr → prev, polls DOM state into curr

### Shot Combos (Preserved)

- **Flat**: click only (no direction held)
- **Top Spin**: W + click
- **Slice**: S + click
- **Lob**: A/D + click

### Game Loop

- `requestAnimationFrame(loop)` replaces `TIC()`
- Delta time tracked but game logic runs at fixed step (1 frame per rAF, ~16.6ms)
- `input.update()` called at top of each frame

### Module Translation Pattern

Each Lua table becomes a JS module exporting an object. Functions map directly:

```lua
-- Lua
ball = {}
function ball.new() return {...} end
function ball.update(b, dt) ... end
```

```js
// JS
export const ball = {
  new() { return {...} },
  update(b, dt) { ... }
};
```

### Menu / Game Over ASCII Art

- ASCII art strings from Lua `print()` calls converted to multi-line strings
- Rendered with `fillText()` line by line at computed positions
